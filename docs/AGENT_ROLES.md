# Agent 角色设计

本系统采用角色分工模式。每个 Agent 只负责一类任务，避免所有 Agent 都什么都做，导致职责混乱。

## 1. Orchestrator Agent

总控 Agent，负责调度整个流程。

### 职责

- 理解用户需求
- 判断是否值得做
- 要求用户确认危险操作
- 调用 Planner 拆任务
- 选择执行 Agent
- 控制任务状态流转
- 触发审查和测试
- 决定是否返工
- 生成最终交付报告

### 不应该做

- 不应该直接乱改代码
- 不应该自动合并 PR
- 不应该跳过 Review
- 不应该在需求不明确时强行执行

## 2. Planner Agent

计划 Agent，负责把用户需求拆成任务。

### 输入

- 用户需求
- 项目背景
- 技术栈
- 当前限制

### 输出

- 是否值得做
- 方案 A / B / C
- 推荐方案
- 风险分析
- 任务拆分
- 每个任务的验收标准

### 输出格式建议

```markdown
# 任务计划

## 需求
...

## 是否值得做
...

## 可选方案
...

## 推荐方案
...

## 风险
...

## 任务拆分
| 任务 ID | Agent | 任务 | 验收标准 |
|---|---|---|---|
```

## 3. Worker Agent

执行 Agent，负责完成具体任务。

可以根据任务类型进一步分为：

- Backend Worker
- Frontend Worker
- Data Worker
- Docs Worker
- DevOps Worker

### 输入

- 任务描述
- 约束条件
- 验收标准
- 相关文件
- 项目规范

### 输出

- 执行摘要
- 修改建议或代码变更
- 修改文件列表
- 自测结果
- 风险说明

### 约束

- 只能做分配给自己的任务
- 不要顺手改无关模块
- 遇到不明确需求要标记 blocked
- 不要删除文件，除非用户明确确认

## 4. Reviewer Agent

审查 Agent，负责检查任务结果。

### 审查维度

- 是否满足验收标准
- 是否有明显 bug
- 是否有边界情况遗漏
- 是否有安全问题
- 是否影响其他模块
- 是否过度设计
- 是否需要返工

### 输出格式建议

```json
{
  "passed": false,
  "issues": [
    {
      "severity": "medium",
      "message": "没有处理空输入。",
      "suggestion": "增加空输入判断和测试。"
    }
  ],
  "required_fixes": [
    "处理空输入",
    "增加对应测试"
  ]
}
```

## 5. Tester Agent

测试 Agent，负责设计测试用例和运行测试。

### 职责

- 生成测试计划
- 运行测试命令
- 分析测试日志
- 识别失败原因
- 给出修复建议

### 测试类型

- 单元测试
- 接口测试
- 边界测试
- 大文件测试
- 脏数据测试
- 安全测试
- 回归测试

## 6. Reporter Agent

汇总 Agent，负责生成最终报告。

### 输出内容

- 本次需求
- 完成内容
- 任务列表
- 修改文件
- 测试结果
- Review 结果
- 剩余风险
- 下一步建议

### 输出格式

优先生成 Markdown，方便保存到仓库、Issue 或 PR 评论中。

## 7. Docs Agent

文档 Agent，负责维护 README、使用说明和开发文档。

### 职责

- 更新 README
- 编写接口说明
- 编写部署说明
- 编写开发流程
- 编写变更记录

## 8. DevOps Agent

部署和运维 Agent。

### 职责

- 编写 Dockerfile
- 编写 docker-compose.yml
- 编写 GitHub Actions
- 设计部署流程
- 检查环境变量

### 高风险操作

以下操作必须用户确认：

- 部署到线上
- 修改生产数据库
- 删除远程资源
- 修改 CI/CD 权限

## 9. 推荐分工原则

1. 一个 Agent 一次只做一种角色。
2. 任务必须有验收标准。
3. Worker 做完必须经过 Reviewer。
4. 代码任务必须经过 Tester。
5. 合并 PR 必须由用户确认。
6. 不要让多个 Agent 同时修改同一个文件。
